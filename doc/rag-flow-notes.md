# RAG 流程笔记

这个项目现在用的是一个轻量版 RAG 方案，目标很简单：

- 读取本地知识文件
- 把长文档切成小片段
- 把每个片段转成向量
- 存到内存向量库
- 用户提问时检索最相关的片段
- 让大模型基于这些片段回答

## 为什么会说“没有相关信息”

现在这个项目里的助手是“知识库优先”的。

如果你问的问题，在 `knowledge/` 目录里的文件中找不到对应内容，模型就会按照提示词直接说：

> 根据您提供的资料，其中没有关于该主题的信息。

这不代表模型坏了，而是说明：

- 知识库里没有相关资料
- 当前配置要求模型优先依据知识库回答

## 这个项目里的 RAG 流程

1. Spring Boot 启动。
2. 扫描 `knowledge/` 目录下的 `.md` 或 `.txt` 文件。
3. 使用 `DocumentSplitters.recursive(700, 100)` 把文档切成小片段。
4. 用 `OpenAiEmbeddingModel` 把每个片段转成向量。
5. 把“向量 + 原文片段”存入 `InMemoryEmbeddingStore`。
6. 用户发来问题时，也把问题转成向量。
7. `EmbeddingStoreContentRetriever` 找出最相似的几个片段。
8. 把这些片段交给聊天模型。
9. 聊天模型生成最终回答。

## 向量数据存到哪里

当前实现里，向量数据存在：

- `knowledge/` 里的原始文档：保存在磁盘
- 分片后的向量和文本：保存在 JVM 内存里

也就是说，这个项目现在用的是：

- `InMemoryEmbeddingStore`

所以它有一个特点：

- 服务重启后，向量数据会消失
- 启动时会重新读取 `knowledge/` 并重新构建

## 为什么分片可以是本地算法

“切分文档”这一步不需要大模型参与。

它只是本地字符串处理，比如：

- 按字符切
- 按段落切
- 按句子切
- 递归组合切

所以分片不需要秘钥，也不需要调用云端模型。

真正需要模型的是：

- `Embedding`：把文本变成向量
- `Chat`：根据检索到的内容组织答案

## 当前项目的行为特点

这版项目更像一个“知识库问答助手”：

- 知识库里有内容，就优先用知识库回答
- 知识库里没有，就明确说没有相关资料

这就是为什么你问一些不在 `knowledge/` 里的内容时，它会拒绝瞎答。

## 后续可以增强的方向

- 把 `InMemoryEmbeddingStore` 换成持久化向量库
- 知识库为空时允许通用问答兜底
- 加 metadata 过滤
- 针对不同文档设置不同切分规则
- 回答里附带来源
