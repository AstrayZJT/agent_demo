## v0.6.0

### 当前版本概览

这一版把 RAG 从“启动就重复切片入库”的学习实现，调整成了更接近企业做法的增量索引模式：

- 直接使用本机 PostgreSQL + pgvector
- 首次启动时全量构建向量索引
- 后续启动时根据文件内容 hash 判断是否变化
- 变更的文档只删旧向量、再写新向量
- 没变化的文档直接跳过，不再重复切片

---

### 本版优化点

#### 1. 切换到本机 pgvector

现在 RAG 直接复用本机 `agentdemo` 数据库中的 pgvector 扩展，不再依赖单独的 Docker 向量库。

优化结果：

- 本地调试更直接
- 数据库结构更少一层
- 更贴近真实项目里“业务库 + 向量扩展”共用一套 PostgreSQL 的做法

#### 2. 改成首次全量、后续增量同步

项目新增了知识库索引表，用来记录每个源文件的 `sourcePath` 和 `contentHash`。

启动时会：

- 读取 `knowledge/` 下的文档
- 计算文件内容 hash
- 对比数据库里的历史索引记录
- 没变化的文档直接跳过
- 变更的文档先删旧向量，再重新向量化
- 已删除的文档同步清理旧向量和索引记录

优化结果：

- 不会每次启动都重复切片入库
- 向量库不会越来越脏
- 更接近生产环境里的增量索引流程

#### 3. 新增索引状态表

这版新增了一个 JPA 索引表，用来保存知识库文件的同步状态，方便判断哪些文档需要重建。

优化结果：

- 文件级别去重更简单
- 后续要做手动重建、定时同步都方便扩展

### 本版的运行特点

#### RAG 路线

请求进入后：

1. 应用启动时先做增量同步
2. 读取 `knowledge/`
3. 计算文件 hash
4. 对比索引表
5. 只处理新增或变更的文档
6. 切片、向量化、写入本机 pgvector
7. 在线请求时再做检索
8. 把检索到的片段交给 LLM

---

### 如何验证这版优化

- 新增一个知识文档后重启，看是否只新增对应向量
- 修改一个知识文档后重启，看旧向量是否被替换
- 不改任何文档时重启，看是否直接跳过索引

---

### 后续建议

- 给索引表加文档大小、更新时间等辅助字段
- 后面如果知识库再变大，可以把入库动作改成定时任务
- 如果需要更强控制，可以再加“手动重建索引”接口

### 本版的运行特点

#### RAG 路线

请求进入后：

1. 读取 `knowledge/`
2. 切片
3. 向量化
4. 检索相关内容
5. 把检索到的片段交给 LLM

#### 工具调用路线

请求进入后：

1. 模型决定是否调用 `webSearch` / `visitWebpage`
2. 工具执行并返回结果
3. 模型基于工具结果组织回答

#### 记忆路线

请求进入后：

1. 根据 `userId` 读取历史消息
2. 拼接上下文
3. 交给模型
4. 回答后再写回数据库

---

#### 如何验证这版优化

- 测 RAG：传 `useRag=true`
- 测工具调用：传 `useRag=false`
- 测匿名请求：传 `userId=-1`
- 测护栏：输入明显注入式内容，看是否被拦截
- 测记忆：同一个 `userId` 连续问两次，看上下文是否延续
